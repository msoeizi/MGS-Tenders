I have made reasonable assumptions where needed and marked open items at the end of each section. # MGS Tender Analysis and Estimating Platform

## Master Architecture Document

### Version 0.1 Discovery Draft

## 1. Product purpose

This product is a web-based estimating workspace for Modern Grain Studios. Its job is to accept one or more tender document uploads, analyze them for commercial millwork relevance, extract project and scope intelligence, generate an evidence-based draft millwork and finish schedule, and convert that into an editable estimating environment that produces both an internal costing view and a client-facing estimate. The system must support iterative improvement during the life of a project rather than rerunning the project from scratch each time. 

The platform is not a static estimator. It is a hybrid review engine with four actors:

1. User
2. GPT analysis layer
3. Python application layer
4. Persistent project state

The product must support two entry paths:

* user starts in ChatGPT and triggers project creation
* user starts in the web app and uploads files there

In both cases, the system must converge to the same project record and the same downstream workflow.

## 2. Product goals

The application is being built to reduce manual tender review time, improve estimating consistency, create auditability around scope interpretation, and allow iterative refinement without losing prior work.

Primary goals:

* detect and classify tender documents
* determine whether the project is relevant to MGS millwork scope
* identify whether scope is install-only, supply-and-install, or mixed/unclear
* preserve an existing usable millwork schedule when present
* construct a millwork schedule when no schedule exists
* extract or build a finish schedule
* prefill estimate line items, labor assumptions, and wording
* preserve evidence and confidence for all major outputs
* let the user edit, override, regroup, and finalize the estimate
* generate client-facing estimate outputs from template-controlled presentation rules

Secondary goals:

* allow supplemental files to improve an existing project rather than replace it
* support row-level or item-level re-review by GPT
* log repeated mistakes, corrections, and rule ideas for later system improvement

## 3. Operating principles

These rules govern the entire application and should be treated as product law.

### 3.1 Drawings override specs

Architectural drawings are the primary authority for millwork scope. Specs, narratives, brochures, or supplementary documents may confirm or clarify, but may not override drawings. 

### 3.2 Demolition drawings do not define new work

Demolition plans must not be used to infer new millwork scope. 

### 3.3 Evidence-based interpretation only

The system must not invent scope because something is common or likely. Every significant conclusion must point to evidence. 

### 3.4 Preserve clean schedules

If a clean, usable millwork or casework schedule exists in the drawing set, that schedule remains the base rather than being reconstructed from scratch. 

### 3.5 Functional-item logic

Rows should represent meaningful functional scope, not tiny internal components. A kitchen, vanity, reception desk, wall panel system, or washroom millwork grouping is a valid line item. Toe kicks, fillers, single gables, and single edges are normally nested under a parent item unless explicitly needed.

### 3.6 Progressive refinement, not destructive reruns

When new files are added or comments are made on specific items, the system should update only impacted portions of the project where possible. It must not wipe and rebuild the entire estimate unless the user explicitly requests a full regeneration.

### 3.7 User override supremacy

The user must always be able to move rows, rename items, split parents into subitems, merge rows, lock values, replace wording, and override scope conclusions. The system assists; it does not own the final answer.

## 4. High-level architecture

## 4.1 Core architectural pattern

The application should be designed around a single source of truth called the Project State. GPT and Python do not directly “own” the project. They both read from and write to the Project State.

### Main components

1. Web frontend
2. Python backend/API
3. Document processing service
4. GPT orchestration layer
5. File storage
6. Database
7. Export engine
8. Rules/debug knowledge area

### Conceptual flow

Files uploaded → project created → documents indexed → GPT triage and extraction → structured results written to Project State → Python builds editable estimating tables → user edits → GPT targeted review actions if needed → final estimate output and PDF export.

## 4.2 Responsibility split

### GPT responsibilities

* classify uploaded documents
* infer project type
* infer scope relevance
* classify install-only vs supply-and-install vs mixed/unclear
* preserve or build millwork schedule
* preserve or build finish schedule
* draft SOW wording
* propose grouping and parent/subitem relationships
* explain evidence
* assign confidence
* re-review selected items on demand

### Python responsibilities

* file upload and storage
* page rendering and text extraction pipeline
* central project state management
* editable UI
* row and table calculations
* labor and material pricing logic
* subcontractor cost blocks
* template builder
* PDF estimate generation
* versioning, audit logs, and persistence
* user permissions and project saving
* delta tracking when supplemental files are added

That split mirrors your stated architecture and should remain clean through the build.

## 10. GPT document intelligence workflow

## 10.1 Stage A: package triage

Inputs:

* file metadata
* per-page text
* page images or page snippets
* prior project context if any

Outputs:

* project candidate title
* project type
* likely relevant files
* likely relevant pages
* likely discipline groupings
* relevance score to MGS scope

## 10.2 Stage B: project summary extraction

Required outputs:

* title
* address
* city/province
* issue date
* due date if found
* designer or consultant info
* building type
* project summary description
* tender discipline
* likely union/non-union if evidence exists

## 10.3 Stage C: scope classification

Determine:

* install only
* supply and install
* mixed/unclear

The weighted evidence model should follow the priority already defined in the brief:
highest weight to responsibility schedules and explicit supply/install notes, medium weight to schedule and detail structure, lower weight to scattered notes and specs. 

The output must include:

* decision
* confidence
* evidence summary
* contradictory evidence if any

## 10.4 Stage D: millwork schedule extraction

Extraction hierarchy:

1. existing schedule
2. detail sheets and millwork detail pages
3. enlarged plans and callouts
4. elevations and room context

If an existing schedule exists, duplicate it first and enrich it second.

If no existing schedule exists, GPT must construct one using the evidence ladder above.

The output per item should include:

* item_id
* name
* room/area
* item tags
* finish codes if visible
* source evidence
* confidence

This remains aligned with your stated millwork schedule logic. 

## 10.5 Stage E: finish schedule extraction

Build or extract a finish schedule with:

* material code
* material category
* description
* areas used
* unit type
* placeholder unit cost

Detect materials like melamine, laminate, veneer, quartz, solid surface, metalwork, mirror, glass, and upholstery. 

## 10.6 Stage F: wording generation

GPT should draft:

* row-level SOW text
* client-facing wording suggestions
* exclusions or assumptions text when requested

## 10.7 Stage G: targeted secondary actions

Inside a project, GPT must support narrow follow-up actions:

* re-evaluate selected item
* explain evidence for selected item
* incorporate supplemental files
* revise wording only
* regroup or split row recommendation

These actions should be scoped, additive, and non-destructive.

## 11. Millwork schedule logic in detail

## 11.1 Row-generation rule

A row becomes its own millwork item when it represents a meaningful scope unit from an estimating or client presentation perspective.

Examples:

* kitchen millwork
* washroom millwork
* boardroom credenza
* reception desk
* branding wall
* guestroom closet
* vanity group

Not separate rows by default:

* toe kick
* single filler
* one side panel
* one edge treatment
* one hinge set

## 11.2 Parent/subitem rule

Create parent items when:

* multiple meaningful components belong to one area
* the scope is conceptually one object but contains separable cost components
* detail density is high
* GPT is uncertain and wants to preserve finer structure
* the user may want to show one row externally and many rows internally

Create subitems when:

* cabinetry and countertop should be priced independently
* glass, stone, metal, or specialty work differs in costing method
* different room components belong under one area heading
* repeated but related components need grouping

## 11.3 Room-based grouping

By default, parents should usually be room or area based unless the item is clearly project-wide, repeated, or generic.

Examples:

* Washroom Millwork → vanity, mirror, supports
* Kitchen Millwork → cabinetry, island, countertop
* Boardroom Feature Wall → panels, integrated metal, glass shelf

## 11.4 Repeat item handling

When a small scope repeats many times across rooms:

* allow quantity-based grouping
* preserve room distribution in evidence or notes
* avoid cluttering the visible estimate with dozens of identical rows unless user chooses detail mode

## 12. Estimating engine logic

## 12.1 Internal estimate table

The internal estimate table should be the primary calculation surface.

Required columns:

* item/row label
* SOW
* material
* hardware
* design hours
* fabrication man × hours
* install man × hours
* item unit price
* quantity
* line total
* misc items/costs
* subcontractor fields where applicable

Derived calculations:

* fabrication cost = fabrication headcount × fabrication hours each × general labor rate
* installation cost = install headcount × install hours each × general labor rate
* design cost = design hours × designer rate
* PM cost may be added if later activated
* line total = unit price × quantity or calculated row logic depending on structure

## 12.2 Labor rates

Initial baseline from the brief:

* designer: 180/hr
* project manager: 120/hr
* general labor: 95/hr 

These should be system-configurable defaults, not hard-coded forever.

## 12.3 Pricing method flags

Each row should carry a pricing method:

* direct formula
* finish schedule driven
* subcontractor driven
* user entered lump sum
* mixed/manual

This helps auditability.

## 12.4 Misc items

Each row should support additional misc cost entries for:

* freight
* special wrapping
* site conditions
* access
* specialty consumables
* coordination
* other ad hoc items

Version 1 can start with one misc amount and one misc note per row. Version 2 may allow multiple misc subentries.

## 12.5 Parent/subitem calculations

If a parent has subitems:

* parent calculation fields become read-only
* parent total = sum of children
* client-facing display may still show the parent only

If a row has no children:

* it remains directly editable

This reflects your required nested estimating behavior. 

## 13. Subcontractor logic

When the system identifies stone, quartz, solid surface, postformed countertops, metalwork, glass, or mirror, it should automatically enable a subcontractor calculation block for that row.

For each applicable row, support:

* subcontract material
* subcontract fabricate
* subcontract install
* supplier note
* quote reference
* inclusion note

This avoids forcing all specialty work through cabinetry logic.

Open questions:

* Should subcontractor totals roll into material only, or display separately in internal view?
* Should certain subcontractor types have default templates?

## 15. Evidence, confidence, and explainability

Every major extracted object should have an evidence drawer or popup.

For each item, the evidence viewer should show:

* source file
* page number
* screenshot snippet if available
* source text excerpt
* explanation of why the item was identified
* confidence score
* source type, such as schedule match, detail clue, elevation clue, finish link, explicit note, or user correction

This is essential to satisfy trust and auditability. The brief explicitly requires preservation of what was detected, why, where it came from, and how confident the system is.

## 25. Executive summary for development handoff

This application should be built as a persistent estimating workspace, not as a one-time AI extraction tool. The center of the system is a shared Project State that both GPT and Python operate against. GPT is responsible for understanding and interpreting tender documents; Python is responsible for managing the estimating environment, calculations, templates, and persistence. The user remains in control at all times through editing, regrouping, override, and targeted re-analysis. The platform must preserve evidence, confidence, and version history so that each estimate is explainable and improvable over time.

## AntiGravity’s responsibility regarding GPT Actions

AntiGravity needs to clearly understand that this product is being built with **GPT Actions in mind**, even if those Actions are not implemented by AntiGravity directly.

The AI side of this program is not a single one-time prompt.
It is expected to include:

### A. Primary GPT Action

This is the main first-pass analysis action.

Its role is to:

* inspect uploaded tender documents
* classify document/package type
* extract project summary
* determine install-only vs supply-and-install vs mixed
* identify relevant sheets
* preserve or build millwork schedule
* preserve or build finish schedule
* prefill wording and draft structured estimate content
* return confidence and evidence references

### B. Secondary GPT Actions

These are follow-up actions used inside an existing project workspace.

Examples include:

* re-evaluate a selected item
* explain evidence for one row
* revise wording for one line item
* regroup or split a row
* incorporate supplemental files
* re-review a specific room or area
* compare user edits against existing GPT output
* improve low-confidence or flagged outputs

These secondary actions are essential to the product concept.
The system is meant to improve incrementally throughout the estimating process, not restart from scratch every time.

So AntiGravity must design the application in a way that supports:

* scoped follow-up actions
* row-level and section-level context
* non-destructive updates
* project-state-based re-analysis
* delta changes instead of full reruns

## What AntiGravity must understand about the GPT side

Even if AntiGravity does not directly configure the GPT Actions, it must understand the program architecture well enough to support them.

The app must be built so that GPT can interact with structured project data cleanly.

That means AntiGravity must design:

* the project state format
* the containers/sections in the project workspace
* the item identifiers
* evidence linking
* row IDs
* review flags
* comments
* snapshot logic
* API structure or service boundaries
* re-analysis request payloads

If these are poorly designed, GPT follow-up actions will become messy and unreliable.

So AntiGravity should assume that the web app is the **system of record**, while GPT Actions act as an **analysis and interpretation layer** working against that system.
